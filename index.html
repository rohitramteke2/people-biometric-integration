Developer Documentation · Plugin Method

Biometric attendance, synced into Zoho People

Deploy the ZAttendance.war plugin on the client’s server. It runs on Apache Tomcat 9, reads punches from the biometric database, and pushes them into Zoho People over OAuth2 — no manual check-ins.

Tomcat 9 + Java JRE Port 8584 MS SQL · MySQL · Access v1.0
client-server — bash

00Prerequisites

Confirm before you start — setup runs on the client’s server.

i
Do this with the client. The integration needs access to the client’s server, the biometric database credentials, and an admin Zoho People account. Keep the client’s IT team or biometric vendor available throughout.
RequirementDetail
Server accessAdministrator access to the client’s Windows server where biometric data is stored
Admin Zoho accountA Zoho People account whose Role is marked admin — OAuth tokens must be generated from it
DatabaseMicrosoft SQL Server, MySQL, or Microsoft Access, with the attendance table reachable
Data centerIndia DC — every Zoho endpoint uses the .in domain

01Install the software

Java first, then Tomcat — the order matters.

Two components go onto the client’s server, in this exact order. Tomcat needs Java already present to register itself correctly.

Java JRE (latest version)

  1. Download the latest build for the server’s OS architecture from OpenLogic OpenJDK.
  2. Run the installer with default options.
  3. Note the install path under C:\Program Files\OpenLogic\jdk-<version> — you’ll point Tomcat at it next.

Apache Tomcat 9 (Windows Service Installer)

Download the Windows Service Installer build from tomcat.apache.org, then set these values during install:

SettingValue
Service nameTomcat9ZohoPeople
Shutdown port8989
Connector port8584 — confirm it isn’t already in use
Java pathC:\Program Files\OpenLogic\jdk-<version>

Set the service to start automatically:

  1. Open Services → find Apache Tomcat 9.0 Tomcat9ZohoPeople.
  2. Right-click → Properties → set Startup type to Automatic → Apply.

02Deploy the ZAttendance plugin

Drop the WAR in the right folder — Tomcat does the rest.

Download the plugin WAR:

urlZAttendance.war
https://www.zoho.com/people/downloads/ZAttendance.war

Place it in the correct folder

The WAR must go into the Tomcat webapps directory — exactly here:

path
C:\ > Program Files > Apache Software Foundation > Tomcat9 > webapps
  1. Copy & paste ZAttendance.war into the webapps folder.
  2. Refresh the folder. Tomcat auto-extracts the WAR and creates a ZAttendance folder beside it — no installer to run.
!
Folder didn’t extract? Make sure the Apache Tomcat 9.0 Tomcat9ZohoPeople service is running. If the ZAttendance folder still doesn’t appear after a refresh, restart the service from Services.

Load the plugin page

Open this URL in a browser on the server to confirm deployment:

verify
http://localhost:8584/ZAttendance

If the page loads, deployment succeeded. The port 8584 must match the Tomcat connector port from Step 01.

!
Plugin page not loading on 8584? The port may be in use by another service. Change it in the config file inside the ZAttendance folder (under Tomcat9 > webapps > ZAttendance), update the Tomcat connector port to match in Tomcat9 > conf > server.xml, restart the Apache Tomcat 9.0 Tomcat9ZohoPeople service, then reload at http://localhost:<new-port>/ZAttendance.

03Configure the plugin

Click Configure, pick the database, fill the fields.

On the plugin page, click Configure in the top-right corner to open the configuration page, then select the database type.

i
Use the example URL from the field itself. Each connection-URL field shows an example in its label / placeholder. Copy that URL into the field and only change the database name and port — don’t type the URL from memory.

Database connection

DatabaseDefault portNotes
MySQL3306Default username root
MS SQL Server1433Enable TCP/IP on 1433 first (below)
MS Access—Give the DB path; switch / to \. Username admin
  1. Paste the example connection URL shown in the field.
  2. Update the database name in the URL.
  3. Set the port to match your database instance.
  4. Enter the database username and password.
  5. Set Delay Time and Sleep Time to 5 minutes (minimum recommended).

MS SQL — enable TCP/IP on 1433

config
Computer Management > SQL Server Network Configuration
  > Protocols for SQLEXPRESS > TCP/IP
  > IP Addresses > TCP Port: 1433 > Apply > OK

04OAuth tokens & SQL query

Generate credentials on the India console, then set the pull query.

!
Data center is India. Use api-console.zoho.in (not .com) and select the India DC when generating the refresh token. Tokens must come from an account whose Role is admin.
  1. Go to api-console.zoho.in and create a Self Client application.
  2. Copy the Client ID and Client Secret into the matching fields on the Configure page.
  3. Under Generate Code, generate a code with the values below.
  4. Paste the Code into the plugin, select India (.in), click Generate, then paste the resulting Refresh Token into its field.
FieldValue
ScopeZohoPeople.attendance.all
Duration10 minutes

Required fields — what each column must return

Every query must return these aliases. The plugin reads them by name, so the aliases matter even when the source columns are named differently.

FieldMeaningFormat
employeeIdMust match the employee ID in Zoho People (or be mapped under Biometric ID Mapping)Text / number
isCheckin1 = check-in (IN punch) · 0 = check-out (OUT punch)1 or 0
eventTimeThe in-punch / out-punch timestampdate-time
downloadDateWhen the record was pushed to the DB (optional — prevents data loss if present)date-time
i
isCheckin direction. Zoho reads 1 as a check-in and 0 as a check-out. If the source table has no direction column, push each punch as both — see the eTimeTrackLite query below.

SQL query — standard (all columns present)

sqlstatic table
select employeeId, isCheckin, eventTime, [downloadDate]
from [ZAttendance].[dbo].[Attendance]
where downloadDate >= $1 and downloadDate < $2

When downloadDate is missing

Set Delay Time to 1–2 hours and window on eventTime instead:

sqlno downloadDate
select employeeId, isCheckin, eventTime
from [ZAttendance].[dbo].[Attendance]
where eventTime >= $1 and eventTime < $2

Production query — eTimeTrackLite (essl)

This is the working query for an eTimeTrackLite database. The AttendanceLogsV1 table has no IN/OUT direction column, so each punch is pushed as both IN ('1') and OUT ('0') via a union. Paste this into the config editor and adjust the table / column names to match the client’s schema.

sqleTimeTrackLite · push each punch as IN & OUT
select EmployeeCode as employeeId, '1' as isCheckin, LogDateTime as eventTime
from [etimetracklite1].[dbo].[AttendanceLogsV1]
where LogDateTime >= $1
and LogDateTime < $2

union

select EmployeeCode as employeeId, '0' as isCheckin, LogDateTime as eventTime
from [etimetracklite1].[dbo].[AttendanceLogsV1]
where LogDateTime >= $1
and LogDateTime < $2
i
Column mapping. EmployeeCode → employeeId, LogDateTime → eventTime. Zoho needs the aliases employeeId, isCheckin, and eventTime — keep these alias names even if the source columns differ.

Time zone

Set the plugin time zone to the time zone of the data in the database (e.g. GMT+5:30). It must match the admin account that generated the tokens and the biometric users’ time zone.

05Verify & test

One employee first, then automate.

Before automating, test with a single employee and confirm the punch lands in Zoho People. Temporarily filter the query:

sqlsingle-employee test
select employeeId, isCheckin, eventTime, [DownloadDate]
from [ZAttendance].[dbo].[Attendance]
where eventTime >= $1 and eventTime < $2 and employeeId = '1001'
  1. In the Manual Sync tab, click Get Record Count to confirm rows are found.
  2. Click Fetch then Upload to pull and push the data into Zoho People.
  3. Confirm the punch under Attendance > My Data / Team Data.
  4. Restore the production query and click Resume Sync to automate periodic pushes.

Biometric ID mapping

  • If a device’s biometric ID differs from the Zoho employee ID, map it under Operations > Attendance > Biometric ID Mapping.
  • If employees have multiple shifts, every shift must be mapped.

Back up the configuration

  • Create a Backup folder inside Apache Software Foundation.
  • Copy the .config file (type .xml) from the Tomcat folder into it.
!
Server stability. The integration depends entirely on the server holding the biometric data. An improper shutdown or crash can corrupt the configuration page and force a reconfigure — keep the .config backup safe.

!Three critical checks

The most common causes of a failed setup.

CHECK 01

ZAttendance port

The plugin port 8584 must match the Tomcat connector port and be free on the server.

CHECK 02

.com → .in (3 places)

In the config properties file, change the domain to .in from .com in all three places it appears.

CHECK 03

Logs & database

Check the plugin / Tomcat logs, and cross-check that the database name and table in the query exist and return rows.

!
Config properties file — change domain in 3 places. Open the config properties file inside the Tomcat ZAttendance folder and replace .com with .in in all three occurrences. Missing even one sends token / API calls to the wrong data center.

06Common issues & fixes

Match a Catalina log line or symptom to its fix.

The plugin’s errors surface in the Tomcat Catalina logs. Expand a group and find the message you’re seeing — or hit ⌘K and paste the error.

Plugin page & Tomcat 6
Plugin page won’t load at :8584/ZAttendance
Tomcat service running? Port 8584 free and matching the connector port? WAR extracted in webapps?
Config page empty; both Resume Sync & Stop Sync show; Configure won’t open
Plugin page corruption (often after an improper server shutdown). The configuration must be reconfigured — restore from your .config backup if you have one.
Address already in use / JVM_Bind Exception
Port 8584 (or shutdown port 8989) is taken by another service. Free the port or change the connector port, then restart Tomcat.
Tomcat page not loading
Tomcat version is greater than 9/10. Use Tomcat 9 — higher versions are not supported for this plugin.
Resume Sync / Configure not working; sync stops; page won’t load
Partial page corruption or a stopped Tomcat service. Restart the Apache Tomcat 9.0 Tomcat9ZohoPeople service; reconfigure if the page is corrupted.
GC overhead limit exceeded
Tomcat ran out of heap, usually from too wide a sync window. Reduce the date range / increase Tomcat memory, then re-sync in smaller windows.
Authentication & tokens 3
Tokens invalid / Access token not found
Wrong DC or expired/incorrect credentials. Regenerate on api-console.zoho.in, select India DC, confirm the domain is .in in all 3 places, and that tokens came from an admin-role account.
JSONObject: Access Token Not Found after Fetch & Upload
Refresh token missing or invalid in the config. Re-paste a freshly generated refresh token and retry.
Exception @getLastUpdatedTime{0}
The plugin can’t read the last-sync marker, typically a token/DC mismatch. Re-verify the tokens and DC as above.
Database & connectivity 5
No suitable driver found for …
Wrong database type selected, or the JDBC connection URL is malformed. Re-select the DB type and paste the example URL from the field, editing only the DB name and port.
Cannot open database "name" requested by the login. The login failed
Bad DB name or credentials. Verify the database name in the query / URL and the username & password.
Connection Refused: Connect / Java socket error
DB not reachable on the configured host/port. For MS SQL, enable TCP/IP on port 1433; confirm the DB service is running.
Communications Link Failure
Network/port issue between plugin and DB. Check the host, port (MySQL 3306 / MS SQL 1433), and that the DB accepts remote connections.
Communications Link Failure + SSLHandshakeException: No Appropriate Protocol
TLS mismatch with the DB. Adjust the connection URL’s SSL parameters (or enable the required protocol) per the DB’s configuration.
Data & sync 6
All record count is 0
Query window returns nothing. Check the table/column names and that data exists in the $1–$2 range.
Data is in the table but record count is 0 — no exceptions in Catalina
Usually a time-zone or window mismatch between the plugin and the data. Align the plugin time zone to the data’s time zone and re-check the date window.
Date range provided is more than 31 days
The sync window exceeds the 31-day limit. Narrow the manual window to 31 days or less.
Attendance skipped / not syncing for some employee IDs
Those IDs aren’t mapped, or the biometric ID differs from the Zoho ID. Map them under Operations > Attendance > Biometric ID Mapping; map all shifts.
No IN/OUT direction column in the table
Push each punch as both IN & OUT — use the eTimeTrackLite UNION query.
downloadDate column not available
Set Delay Time to 1–2 hours and window on eventTime instead. Without downloadDate, full-data delivery isn’t guaranteed — possible data loss.
i
Where to look. Catalina logs live under Tomcat9 > logs > catalina.<date>.log. Open the latest file and search for the message text above.
